# PERF-1012: Unroll buffer type dispatch in multi-worker writer chunk loops

## Focus Area
The multi-worker writer chunk loops in \`CaptureLoop.ts\` (\`while (nextFrameToWrite < chunkEnd)\`). This loop processes frames generated by multiple workers. Currently, it evaluates buffer length and handles stream writes polymorphically via \`(buffer as any).length\` and \`stream.write(buffer as any)\`.

## Background Research
The loop in question handles both Canvas and DOM rendering strategies. Canvas strategies yield instances of Uint8Array/Buffer, and DOM strategies (before decoding) might yield strings, though they are decoded. The \`buffer as any\` forces V8 into a polymorphic inline cache (PIC) state when evaluating \`.length\` and invoking \`.write()\`. As previously identified and explored in \`PERF-987\`, \`PERF-994\` and \`PERF-1011\`, explicit unrolling of this path based on \`isDomStrategyWriter\` maintains monomorphic inline caches (MICs), which improves execution speed.

In the initialization block (\`while (!aborted) { ... }\`), this check has already been explicitly unrolled based on \`isDomStrategyWriter\`:
\`\`\`typescript
            let writeSuccess = false;
            if (isDomStrategyWriter) {
              const buf = buffer as unknown as Buffer;
              pendingBytes += buf.length;
              writeSuccess = stream.write(buf);
            } else {
              pendingBytes += (buffer as any).length;
              writeSuccess = stream.write(buffer as any);
            }
\`\`\`

However, the chunk writer loop below it (\`while (nextFrameToWrite < chunkEnd)\`) still uses the polymorphic version:
\`\`\`typescript
                const buffer = frameBufferRing[ringIndex]!;
                pendingBytes += (buffer as any).length;
                const writeSuccess = stream.write(buffer as any);
\`\`\`

Applying the same unrolling to this loop will prevent polymorphic degradation and improve execution speed for the multi-worker path. (This plan repeats the intent of PERF-987/1011, making sure the specific chunking block is replaced).

## Implementation Spec

### Step 1: Unroll buffer type dispatch in multi-worker chunk processing loop
**File**: \`packages/renderer/src/core/CaptureLoop.ts\`
**What to change**:
In the multi-worker loop under \`while (nextFrameToWrite < chunkEnd) {\`, replace the polymorphic \`pendingBytes\` addition and \`stream.write\` calls:

\`\`\`typescript
                const buffer = frameBufferRing[ringIndex]!;
                pendingBytes += (buffer as any).length;
                const writeSuccess = stream.write(buffer as any);
\`\`\`

With an explicit branch checking \`isDomStrategyWriter\` to enable monomorphic behavior (matching the structure used in the initial frame capture):

\`\`\`typescript
                const buffer = frameBufferRing[ringIndex]!;
                let writeSuccess = false;
                if (isDomStrategyWriter) {
                  const buf = buffer as unknown as Buffer;
                  pendingBytes += buf.length;
                  writeSuccess = stream.write(buf);
                } else {
                  pendingBytes += (buffer as any).length;
                  writeSuccess = stream.write(buffer as any);
                }
\`\`\`
**Why**: By explicitly asserting the type as \`Buffer\` when \`isDomStrategyWriter\` is true, V8 can maintain a monomorphic inline cache for \`.length\` and \`.write()\` in the DOM path, avoiding polymorphic lookups and improving execution speed.
**Risk**: Minimal. This pattern is already proven and used in the single-worker path and the initialization block of this very multi-worker path.
