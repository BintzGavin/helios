# PERF-187: Target Element BeginFrame Parameter Unrolling

## Focus Area
DOM strategy fallback logic when `targetElementHandle` is present in `packages/renderer/src/strategies/DomStrategy.ts`.

## Background Research
Currently, when a `targetSelector` is provided in `DomStrategy.ts`, the renderer calculates the bounding box via `targetElementHandle.boundingBox()` and uses `HeadlessExperimental.beginFrame` with `clip` parameters. However, the entire `capture` function returns a dynamically chained Promise (`.then()`) that allocates multiple closures per frame.

Specifically, the `targetElementHandle` path incurs two consecutive async allocations inside its hot path:
1. `this.targetElementHandle.boundingBox().then(...)`
2. `this.cdpSession!.send(...).then(...)`

Converting the entire `capture` method into an `async/await` state machine rather than manually chaining Promises avoids the V8 overhead of repeatedly allocating anonymous `.then()` closures on every frame. This optimization simplifies the generated byte code and unrolls the inline object parameters cleanly.

## Benchmark Configuration
- **Composition URL**: `file:///app/output/example-build/examples/simple-animation/composition.html`
- **Render Settings**: 1280x720, 30fps, 5 seconds (150 frames)
- **Mode**: dom
- **Metric**: Wall-clock render time in seconds
- **Minimum runs**: 3 per experiment, report median

## Baseline
- **Current estimated render time**: ~3.84s (with targetSelector)
- **Bottleneck analysis**: Micro-stalls from allocating anonymous closures (`.then()`) for Promise chaining inside the hot loop.

## Implementation Spec

### Step 1: Refactor `capture` to use async/await
**File**: `packages/renderer/src/strategies/DomStrategy.ts`
**What to change**:
Change `capture(page: Page, frameTime: number): Promise<Buffer>` to `async capture(page: Page, frameTime: number): Promise<Buffer>`.
Replace the entire `.then()` chains with `await` and return the buffer directly.
