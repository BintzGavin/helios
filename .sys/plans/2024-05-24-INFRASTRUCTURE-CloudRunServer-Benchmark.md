#### 1. Context & Goal
- **Objective**: Implement performance benchmarks for the `CloudRunServer` entrypoint to measure request parsing and handling overhead.
- **Trigger**: The V2 distributed rendering vision requires scalable cloud executions. Benchmarks are an allowed fallback action under the "Nothing To Do Protocol". Knowing the overhead introduced by the `createCloudRunServer` HTTP handling wrapper is important for tuning large-scale job distributions.
- **Impact**: Establishes performance baselines for the HTTP handler inside Google Cloud Run instances. This guarantees that request parsing and routing don't introduce performance bottlenecks compared to the execution logic itself.

#### 2. File Inventory
- **Create**: `packages/infrastructure/tests/benchmarks/cloudrun-server.bench.ts`
  - Purpose: Contains the Vitest benchmark suite for the `CloudRunServer` HTTP handler.
- **Read-Only**: `packages/infrastructure/src/worker/cloudrun-server.ts`

#### 3. Implementation Spec
- **Architecture**: Create a Vitest benchmark suite that measures the throughput of the HTTP server generated by `createCloudRunServer`.
  - Mock `WorkerRuntime.prototype.run` to return instantly so the benchmark isolates the HTTP server's parsing and request routing overhead.
  - Setup the server using a dynamic port in a `beforeAll` hook.
  - Teardown the server in an `afterAll` hook to prevent `EADDRINUSE` errors across benchmark runs.
- **Pseudo-Code**:
  - Mock `WorkerRuntime.prototype.run` to immediately resolve, bypassing actual rendering work.
  - Set up a standard `vitest bench` suite named 'CloudRunServer Benchmark'.
  - In a `beforeAll` hook, instantiate the cloud run server with a dynamic port and start listening.
  - In an `afterAll` hook, close the server instance gracefully to avoid dangling ports.
  - Inside the `bench` block, execute a fast HTTP POST request against the local server port containing a minimal valid JSON payload (`jobPath`, `chunkIndex`).
- **Public API Changes**: None.
- **Dependencies**: None.
- **Cloud Considerations**: This validates the entrypoint's HTTP wrapper overhead inside GCP, independent of the network latency.

#### 4. Test Plan
- **Verification**: `npm run bench -w packages/infrastructure -- tests/benchmarks/cloudrun-server.bench.ts --run`
- **Success Criteria**: The benchmark runs successfully without throwing `EADDRINUSE` or other crashes, and outputs operations per second for the HTTP handling logic.
- **Edge Cases**: Ensure the server cleanly shuts down via `afterAll` to avoid port exhaustion across multiple benchmark runs.
- **Integration Verification**: Verify no existing tests or builds break (`npm test -w packages/infrastructure`).
