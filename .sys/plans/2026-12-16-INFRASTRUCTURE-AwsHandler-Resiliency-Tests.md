3. Implementation Spec
- **Architecture**: The tests will invoke the asynchronous handler function generated by `createAwsHandler` with various malformed event payloads to verify its resiliency. It will mock `WorkerRuntime` to simulate both successful executions and unexpected internal throws, verifying the handler correctly maps these to appropriate AWS API Gateway compatible HTTP status codes without crashing.
- **Pseudo-Code**:
  - Create a test block for `createAwsHandler Resiliency`.
  - Implement a test for an empty or undefined payload event and verify it returns a 400 Bad Request status code.
  - Implement a test for an event payload missing `jobPath` or `chunkIndex` and verify it returns a 400 Bad Request.
  - Mock `WorkerRuntime.prototype.run` to throw a standard Error (e.g., `new Error('Internal fetch exception')`) and verify the handler catches it and returns a 500 status code with the error message.
  - Mock `WorkerRuntime.prototype.run` to return a failed execution (`{ exitCode: 1, stdout: '', stderr: 'Render crash' }`) and verify the handler returns a 500 status code with the stderr embedded in the body.
  - Mock `WorkerRuntime.prototype.run` to throw a non-Error object (e.g., throwing a string or null) to ensure the generic `catch (error: any)` block falls back to `'Unknown error in AWS Lambda handler'`.
- **Public API Changes**: None.
- **Dependencies**: None.
- **Cloud Considerations**: The tests verify the specific contract expected by AWS Lambda Node.js proxy integrations (returning `{ statusCode, body }` objects).
