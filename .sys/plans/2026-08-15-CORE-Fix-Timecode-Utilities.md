# Context & Goal
- **Objective**: Fix `timecodeToFrames` and `framesToTimecode` in `packages/core` to robustly support high frame rates (>99fps) and long durations (>99 hours).
- **Trigger**: Code review revealed that `timecodeToFrames` regex strictly enforces 2 digits, causing failures for valid high-FPS or long-duration inputs generated by `framesToTimecode`.
- **Impact**: Ensures correct timecode handling for all valid inputs, preventing runtime errors and invalid displays in consumer applications like Helios Studio or CLI.

# File Inventory
- **Modify**: `packages/core/src/timecode.ts` (Update regex and padding logic)
- **Modify**: `packages/core/src/timecode.test.ts` (Add regression tests for edge cases)
- **Read-Only**: `packages/core/src/errors.ts`

# Implementation Spec
- **Architecture**: Update regex to allow variable digits for hours and frames, and make padding adaptive.
- **Pseudo-Code**:
  - In `framesToTimecode`:
    - Determine frame padding width: `max(2, floor(log10(fps-1)) + 1)` (e.g., 30fps -> 2, 120fps -> 3).
    - Use `padStart` with calculated width for frames.
    - Hours should default to 2 digits but allow expansion.
  - In `timecodeToFrames`:
    - Update regex from `^(\d{2}):(\d{2}):(\d{2}):(\d{2})$` to `^(\d+):(\d{2}):(\d{2}):(\d+)$` to capture hours and frames of any length.
    - Parse captured groups as integers.
- **Public API Changes**: None (internal logic fix). `framesToTimecode` output format remains compatible for standard cases but extends for edge cases.
- **Dependencies**: None.

# Test Plan
- **Verification**: Run `npm test -w packages/core`
- **Success Criteria**:
  - Existing tests pass.
  - New test `framesToTimecode(100, 120)` returns `"00:00:00:100"`.
  - New test `timecodeToFrames("00:00:00:100", 120)` returns `100`.
  - New test `framesToTimecode(2700000, 30)` returns `"25:00:00:00"`.
  - New test `timecodeToFrames("100:00:00:00", 30)` returns correct frame count.
- **Edge Cases**:
  - FPS = 10, 100, 1000.
  - Duration > 100 hours.
